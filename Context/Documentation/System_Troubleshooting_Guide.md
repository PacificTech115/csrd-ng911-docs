# System Troubleshooting Guide

Common issues and diagnostic steps for the NG911 system, organized by component.

## AI Backend Issues

### AI backend not responding / "Connection Error" in chat
**Symptoms:** Chat shows "Connection Error: Could not reach the AI backend" or thinking dots never resolve.

**Diagnosis:**
1. Check if the NSSM service `CSRD_AI_Backend` is running: `nssm status CSRD_AI_Backend`
2. Check if port 8000 is in use: `netstat -ano | findstr :8000`
3. Check the Cloudflare Tunnel status (routes `ai.pacifictechsystems.ca` to localhost:8000)

**Fix:**
- Restart the NSSM service: `nssm restart CSRD_AI_Backend` (requires admin terminal)
- If port 8000 is held by a zombie process: `taskkill /PID <pid> /F` from an admin terminal, then restart the service

### Port 8000 binding error (WinError 10048)
**Symptoms:** Service fails to start with "error while attempting to bind on address ('0.0.0.0', 8000): [winerror 10048] only one usage of each socket address"

**Diagnosis:** Another process is holding port 8000.

**Fix:**
1. Find the PID: `netstat -ano | findstr :8000`
2. Kill the process: `taskkill /PID <pid> /F` (must be run from an admin terminal)
3. Restart the service: `nssm restart CSRD_AI_Backend`

### AI gives incorrect information
**Possible causes:**
- Knowledge base is out of date (new files added but ingestion not run)
- Question is ambiguous and AI guessed wrong

**Fix:**
- Run re-ingestion: call the `/api/reingest` endpoint or run `python -m database.ingest` from the AI directory
- Ask a more specific question or provide context in your message

## GP Tool Issues

### GP job stuck in "Waiting" or "Executing" for too long
**Symptoms:** Job status shows "esriJobWaiting" or "esriJobExecuting" for more than 30 minutes.

**Diagnosis:**
1. Check ArcGIS Server Manager for the GP service status
2. Look at server logs for errors
3. Verify the SDE connection file is accessible from the server

**Fix:**
- Cancel the job from ArcGIS Server Manager
- Restart the GP service if stuck
- Re-submit the job

### Export tool fails with "target_fc not found"
**Symptoms:** Export GP tool returns error that the feature class doesn't exist.

**Diagnosis:** The target_fc path may be incorrect or the SDE connection has changed.

**Fix:**
- Verify the feature class path: default is `SDE.NG911\SDE.NG911_SiteAddress`
- Check that the SDE connection file (`sde@regional.sde`) is valid and accessible
- Verify the feature class exists in the geodatabase using ArcGIS Pro

### Export tool fails with network share error
**Symptoms:** Export succeeds but ZIP copy to `\\GIS\Scripts\Geoshare\NG911 Exports` fails.

**Fix:**
- Verify the network share is accessible from the ArcGIS Server machine
- Check that the ArcGIS Server service account has write permissions to the share
- The local ZIP is still available in the server's scratch folder as a fallback

### QA validation finds many failures
**Symptoms:** QA report shows high counts of mandatory null failures or NGUID issues.

**This is expected behavior** when:
- New records were added without filling all mandatory fields
- Records were imported from an external source without NGUID generation

**Fix:**
- Review the CSV reports generated by the QA tool for specific records
- Fix the flagged records in the appropriate editing version
- Re-run QA to confirm fixes

### Reconcile fails with conflicts
**Symptoms:** ReconcilePost returns conflict errors, especially with ABORT_CONFLICTS policy.

**Diagnosis:** Two users edited the same record in different versions.

**Fix:**
- Re-run with `conflict_policy = NO_ABORT` to auto-resolve using the configured favor policy
- For MUNI_TO_QA: municipal edits win (FAVOR_EDIT_VERSION)
- For DEFAULT_TO_MUNI_SYNC: DEFAULT wins (FAVOR_TARGET_VERSION)
- Review the reconcile log for specific conflicting records

## Web App Issues

### CMS edits not appearing after save
**Symptoms:** Admin saves content edits but the page still shows old content.

**Fix:**
- Hard refresh the page: Ctrl+Shift+R
- Clear browser cache for `apps.csrd.bc.ca`
- Verify the edit was saved by checking the CMS Hosted Table directly in Portal

### Login fails or token expired
**Symptoms:** Login screen appears, credentials are correct but login fails. Or session expired mid-use.

**Fix:**
- Clear localStorage: open browser DevTools (F12), go to Application > Local Storage, delete entries for `apps.csrd.bc.ca`
- Try logging in again
- If Portal is down, check `apps.csrd.bc.ca/hub/home` directly

### Sync App shows "0 features" or "schema mismatch"
**Symptoms:** Sync App verification succeeds but comparison shows zero features or errors.

**Diagnosis:**
- Token may have expired during the long fetch operation
- Source Feature Service may have changed its schema
- Match field selection may be incorrect

**Fix:**
- Log out and log back in to refresh the token
- Verify the source URL returns features when accessed directly in a browser
- Check that GlobalID field exists in the source and FeatureID exists in the target

### Pages not loading or showing blank content
**Symptoms:** Navigation works but page content area is empty.

**Fix:**
- Check browser console (F12) for JavaScript errors
- Verify the HTML partial file exists in `Web App/docs/partials/`
- If CMS-driven content is blank, the CMS Hosted Table may be unreachable

## Pipeline Issues

### Nightly pipeline did not run
**Symptoms:** Automations Dashboard shows old run date, no email notification received.

**Diagnosis:**
- Check ArcGIS Notebook schedule in Portal
- Verify the Orchestrator notebook (Item ID: `811614c266a84b769c1fe9ffbedda058`) is published and scheduled

**Fix:**
- Re-publish the notebook if it was unpublished
- Manually trigger from the Automations Dashboard "Force Run" button
- Check ArcGIS Server logs for notebook execution errors

### Pipeline shows "success_qa_warnings"
**Symptoms:** Pipeline completed but with QA warnings. Data was still promoted to DEFAULT.

**This is expected** because `QA_BLOCKING=False`. The pipeline continues even when QA finds issues.

**Action:** Review QA results on the Automations Dashboard, fix any flagged records, they will pass QA on the next run.

### Salmon Arm ETL sync issues
**Symptoms:** Salmon Arm data not syncing or showing stale data.

**Diagnosis:**
- Check the ETL notebook (Item ID: `fb8fd369499b440c8ae3720c1bbe3b9f`)
- Verify Salmon Arm's hosted Feature Layer is accessible
- Check the ETL dashboard section on `#automations-dashboard`

**Fix:**
- Re-run the ETL notebook manually from Portal
- Verify the hosted layer URL hasn't changed
- Check the Power Automate webhook for ETL completion notifications
